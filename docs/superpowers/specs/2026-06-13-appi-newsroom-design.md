# APPI Newsroom — 日更內容引擎設計

> 日期：2026-06-13
> 作者：Lightman（CΛ）
> 目標：以 Claude Code 自訂 skill 為寫作引擎，穩定產出「科技」類高品質文章達到日更，同時守住 APPI「可信內容資產」的定位。

## 1. 背景與目標

- APPI News 是「AI 輔助寫作 + 專家審稿 + 媒體刊登」的專業觀點平台。Lightman 負責 tech 類，目前累積 6 篇（4 篇 tech、2 篇 health），皆為深度長文。
- 目標：**至少每天一篇** tech 文章，且不稀釋可信度（不淪為 AI 內容農場）。
- 核心洞見：日更的瓶頸不是「人的產能」，而是「系統設計」。把「定稿」從文章末端搬到**開頭**——由人提供「想要的結論」，AI 負責擴寫與查證——每篇就天生是作者的真人判斷，而非 AI 自說自話。

## 2. 關鍵決策

| # | 決策點 | 結論 |
|---|--------|------|
| 1 | 人的角色 | 人在開頭提供結論（觀點）；AI 負責擴寫、查證、配圖、產出。定稿前移。 |
| 2 | 新鮮度 | 雙引擎：熱點短稿（當天時效）＋ 週級深度稿。 |
| 3 | 議題雷達 | **至少 3 題**候選（不設上限），各附「寫作方向 + 候選結論」；**可一次複選多題**，於同一 session 連續完成。 |
| 4 | 互動方式 | 輸入 `/newsroom` → 自動找相關議題 → **一問一答引導**逐步把稿件完成（不需一開始就給結論）。 |
| 4b | 引用與語言 | 所有資料附 inline 來源超連結、全文超連結皆驗證可連線；全文繁體中文 + 台灣用語。 |
| 4c | 文風 | **去 AI 腔**：禁破折號、禁 AI 套語句式、禁 AI 語氣；寫得像有立場的台灣人（詳見 §4.7）。 |
| 5 | 可信度閘門 | 起草內建「超連結驗證—重寫」迴路：全文每條超連結逐條 fetch，失效/捏造者由 AI 找真實來源替換或改寫論點，迴圈到全綠才產出。無額外人工審稿關卡。 |
| 6 | 實作形式 | 一個 Claude Code 專案級自訂 skill，於使用者現有的 Claude Code session 互動執行，吃現有 Claude 訂閱。 |
| 7 | 節奏 | 手動批次 + 排程：一次 session 批量寫多篇，`status: scheduled`，由既有部署在排程時間到後自動上線。 |
| 8 | 段落配圖 | **每段落依內容用 OpenAI 影像模型（gpt-image）生成對應插圖**，避免閱讀疲勞；人物一律台灣人（沿用既有 `applyPeopleDirective` 鐵律）；須遵守 `PERFORMANCE.md`（lazy-load、webp、依尺寸縮圖、aspect-ratio 鎖 CLS）。 |
| 9 | 排程確認 | 全部文章寫完後，**依議題熱門程度推薦各篇 `publishDate`**，一次列出整批排程表供使用者確認 / 調整，確認後才套用並 push。 |
| 10 | 預覽修改 | 產檔後提供**本地預覽連結**（`pnpm dev` 渲染後的該文章 URL）供閱讀與修改；修訂滿意後才進排程確認與上線。 |

## 3. 架構概觀

整套系統 = **一個 Claude Code skill（`newsroom`）** + 既有 repo 內容結構 + 既有部署流程。無新增服務、資料庫或設定。

使用者輸入 `/newsroom` 後，skill 啟動一段**引導式對話**，全程一問一答把稿件完成：

```
使用者：/newsroom
   ▼
① 議題雷達     掃來源 → 評分 → 列「至少 3 題」候選（附建議方向 + 候選結論）
   ▼          使用者可一次複選多題
   ▼
② 逐題問答     對每一選定題，一次問一題（多選題優先）萃取觀點：
   ▼          核心結論 → 切角/強調面向 → 真人經驗 → 篇幅/讀者 → 指定來源
   ▼
③ 逐題起草     擴寫（每段配 OpenAI 生成圖，人物=台灣人）→ 超連結驗證重寫迴路
   ▼          → check:links → 封面圖 → 完整 frontmatter → 寫成 .md
   ▼          → 提供本地預覽連結供閱讀/修改
   ▼
④ 批次排程     全部寫完 → 依熱門度推薦各篇 publishDate → 列整批排程表供確認/調整
   ▼
使用者：確認排程 → git commit && push（一次推一批）
   ▼
既有部署流程 → GitHub Pages
   （靜態站於排程時間後重建，scheduled 文章到時自動現身；既有機制，本案不改動）
```

## 4. `newsroom` skill 設計

### 4.1 放置位置

專案級 skill：`.claude/skills/newsroom/`。
理由：此 skill 緊密依賴 appi.news 的 content schema、目錄結構、`check:links` 與封面圖流程，應隨 repo 版控、與內容結構同步演進。

**啟動**：使用者輸入 `/newsroom`，skill 進入引導式對話，依序執行 §4.2 → §4.3 → §4.4，不需使用者一開始就提供結論。

### 4.2 步驟一：議題雷達

`/newsroom` 一啟動即自動掃描來源、評分、列出候選題（**至少 3 題，不設上限**）。

**來源**（預設，可於 skill 內調整）：
1. 主流模型發布：Anthropic / OpenAI / Google 官方 blog
2. arXiv `cs.AI` / `cs.CL` 近期熱門
3. Hacker News 高分科技話題
4. 本業交叉題：醫療 AI、健康法規、合規

**評分**：`熱度 × APPI/tech 相關度 × 差異化角度可得性`（三軸各 1–10，相乘排序）。熱度分數同時供步驟四的排程推薦使用。

**輸出**：每個候選題附——
- 議題（發生什麼）
- 為何相關
- **建議寫作方向**（切角）
- **候選結論**（可直接採用或改一句）
- 分數（熱度／相關／差異化）

**選題**：使用者可**一次複選多題**（如「1、3、5」），於同一 session 連續完成；每個選定題各跑一輪步驟二、三。

### 4.3 步驟二：問答引導（完成稿件的核心互動）

對**每個選定題逐一**進行。skill 以**一問一答**陪使用者把該篇稿件輪廓談定——**一次只問一題、可選時用多選題**（沿用 brainstorming 的提問紀律），避免一次丟一堆問題。建議問序：

1. **核心結論**（採候選結論，或改寫成你的判斷）
2. **切角 / 想強調的面向**
3. **真人觀點 / 本業經驗**（要放進去的個人判斷、案例——這是差異化來源）
4. **篇幅與讀者**（深度旗艦 3000+ 字 / 速讀短稿 800–1500 字；目標讀者）
5. **指定必引來源**（可選）

問答結束後，skill 彙整成「結論 + 脈絡 + 角度 + 真人觀點」的起草輸入，進入步驟三。過程中使用者隨時可補充或修正。複選多題時，逐題完成步驟二、三後，再進入步驟四批次排程。

### 4.4 步驟三：起草與產出

依問答結果執行：
1. **查料**：依結論用 WebSearch / WebFetch 蒐集支撐材料與真實來源。
2. **擴寫（資料必附來源超連結）**：繞著結論建立論證結構、寫正文。
   - **凡引用任何數據、事實、研究、引述，正文當下即附該來源的 inline 超連結**，不可只把連結堆在文末。讀者點任一論據都能直達原始出處。
   - 無法附上可驗證來源超連結的資料，**不得寫入**（改寫成不需佐證的論述，或捨棄）。
   - 分層：深度旗艦（給較完整脈絡 + 結論）3000+ 字；熱點 / 速讀短稿 800–1500 字。
3. **超連結驗證—重寫迴路（可信度命脈）**：對**全文每一條超連結**（正文 inline 引用 + 文末 references，無一例外）：
   - 逐條 fetch 驗證：連結可達（HTTP 2xx，確保能連線）+ 內容確實支持文中該句。
   - 無效 / 連不上 / 內容對不上者 → 找真實存在、能佐證同一論點的來源替換；找不到佐證 → 改寫或移除該論點，**不硬掰、不留死連結**。
   - 重驗，迴圈到**所有超連結全部可連線且內容相符**才通過。
4. **站內連結**：跑 `pnpm check:links` 確認站內連結無誤。
5. **封面圖**：產生封面並存至 `public/covers/`（沿用既有 `coverImage: "covers/<slug>.jpg"` 慣例）。
6. **段落配圖（避免閱讀疲勞）— 每段必配**：**每一個段落都依其內容用 OpenAI 影像模型（gpt-image）生成一張對應插圖**（單篇段落圖數量 = 段落數）。
   - 沿用既有生圖管線與 **`applyPeopleDirective` 鐵律**：圖中若出現人物，一律台灣人（East Asian, Han Taiwanese），任何 prompt 都自動附加。
   - 風格與封面一致（簡約編輯插畫、無文字）。
   - **效能硬約束（遵守 `PERFORMANCE.md`）**：所有段落圖一律 `loading="lazy"`、轉 webp、依顯示尺寸縮圖、以 `aspect-ratio` 鎖住版位（CLS 維持 0）；存 `public/images/`。
   - 成本控管不靠減少張數（每段必配為硬需求），改用較低生圖品質（`OPENAI_IMAGE_QUALITY=low`）與 webp 壓縮控制單張成本與大小。
7. **frontmatter**：補齊 `title / description / category / subcategory / tags / highlights / references / author: "lightman" / sourceType: "ai-assisted" / disclaimerType`。
8. **文風複查**：產檔前過一遍 §4.7「去 AI 腔」守則（破折號、AI 套語、AI 語氣），違反項改寫。
9. **產檔**：寫入 `src/content/articles/<slug>.md`，`status: "draft"`（排程待步驟四決定）。
10. **本地預覽連結**：啟動 / 沿用本地 `pnpm dev`，提供該文章**渲染後的預覽 URL** 供使用者**閱讀與要求修改**；修訂於同一 session 即時反映，滿意後才進步驟四。

### 4.5 步驟四：批次排程確認

所有選定題都完成步驟二、三後：
- skill **依議題熱門程度（步驟一的熱度分數）推薦各篇 `publishDate`**：愈熱門 / 愈具時效者排愈前，其餘錯開灑滿後續日期（一天一篇，維持日更）。
- **一次列出整批排程表**（檔名 / 標題 / 建議日期 / 熱度）供使用者**確認或調整**。
- 確認後，套用各篇 `publishDate` + `status: "scheduled"`，一次 `git commit && git push`。

> 即時發佈情境：若某篇要立刻上線，於排程表把它設 `status: "published"` + `publishDate` 現在即可。

### 4.6 differentiation 守則（寫進 skill 的硬性要求）

- **全文一律繁體中文 + 台灣用語**：使用台灣慣用詞（軟體 / 程式 / 網路 / 演算法 / 人工智慧 / 影片 / 數位 / 介面 / 預設…），**禁止**中國用語（軟件 / 程序 / 網絡 / 算法 / 人工智能 / 視頻 / 數字 / 接口 / 默認…）。標題、正文、frontmatter（description、highlights、tags 等）皆適用。
- **生成圖片中若出現人物，一律台灣人**（沿用 `applyPeopleDirective` 鐵律）；段落圖須符合 `PERFORMANCE.md` 效能約束。
- 每篇**必須**有作者真人觀點層（結論即為其種子）；AI 不得自行替換或稀釋使用者給的結論。
- **所有資料（數據/事實/研究/引述）必附來源超連結**，正文 inline 標註，不可只列文末。
- **全文每一條超連結都必須通過 §4.4 step 3 驗證**（可連線 + 內容相符）；無有效佐證的論點不得留下，不得出現死連結。
- 保留 `expertNote` / `risksAndLimits` 的填寫空間，維持與內容農場的分界。

### 4.7 文風守則：去 AI 腔，寫得像人（硬性要求）

產檔前必過一道文風複查，違反項一律改寫。

**禁用標點 / 排版**
- **禁用破折號**（`—` `──` `--`）：改用句號、逗號、冒號，或直接拆句。
- 不濫用粗體與條列：條列只在真的是並列項目時用，論述用段落寫。

**禁用 AI 套語句式**
- 對比升華：「不僅⋯更⋯」「不只是⋯而是⋯」「與其說⋯不如說⋯」。
- 排比堆疊：濫用「從 A 到 B 到 C」、三項式排比、「無論⋯還是⋯」。
- 萬用轉折 / 強調：「值得注意的是」「事實上」「更重要的是」「某種程度上」。
- 宏大開場 / 收尾：「在這個⋯的時代」「總而言之」「歸根結底」「說到底」。
- 自問自答設問腔：「那麼，⋯呢？」式開場。
- buzzword：賦能、解鎖、釋放潛力、顆粒度、閉環、抓手等空話。

**禁用 AI 語氣**
- 過度正向熱情、四平八穩無立場、空泛升華、結尾硬要總結拔高。
- 翻譯腔（英文直譯的長句、濫用「們」、堆連接詞）。

**正面原則**
- 有明確主張與立場（結論是人給的，不得稀釋成中性平衡）。
- 多具體例子、數字、場景，少抽象名詞。
- 句長有變化，可用短句；段落開門見山，不寫鋪墊式導言。
- 寫台灣人說話的語感，自然口語但不鬆散。

## 5. 日常運作流程

**批次寫作 session（每週一次或每幾天一次）**：
1. 輸入 `/newsroom` → 看雷達候選題（≥3 題）→ **複選想寫的幾題**（如 5–7 題）。
2. skill 逐題跑問答引導 + 起草（含段落配圖）→ 每篇給本地預覽連結供你閱讀 / 修改。
3. 全部完成後，skill 依熱門度推薦整批 `publishDate` → 你確認 / 調整排程表。
4. 確認後一次 `git commit && git push`。
5. 文章於各自排程時間自動現身。

**臨時熱點**：當天 `/newsroom` 選一題，問答時選短稿、結論直接給，排程表把它設 `published` + 現在時間，push 即上。

## 6. 範圍界線（YAGNI）

**納入**：`newsroom` skill（雷達複選 + 逐題問答 + 起草 + 超連結驗證迴路）、封面圖串接、**段落配圖（OpenAI gpt-image，沿用 `applyPeopleDirective`）**、本地預覽連結、批次排程確認。

**新增相依（需一併處理）**：
- 段落圖所需的 content schema / Markdown 渲染擴充（目前僅有封面圖，無段落圖機制）。
- 段落圖的效能處理（webp、依尺寸縮圖、lazy-load、CLS 0），對齊 `optimize-home-images.mjs` 的內頁版。
- OpenAI 生圖金鑰（`OPENAI_API_KEY`，沿用既有 worker / `regen-covers.mjs` 設定，不進 repo）。

**排除**：無人值守自動產稿、績效回饋迴路（皆可未來另案）。

## 7. 風險與緩解

| 風險 | 緩解 |
|------|------|
| AI 捏造來源 / 死連結 | 資料必附 inline 來源超連結；§4.4 驗證—重寫迴路逐條 fetch 確認可連線且內容相符，出檔前替換或移除。 |
| 量產稀釋可信度 | 結論由人給 + differentiation 守則 + references 查證。 |
| 訂閱額度與互動寫程式共用 | 批次寫作集中時段進行；篇數實測抓手感。 |
| 段落圖拖垮內頁效能 | 全段落圖 lazy-load + webp + 依尺寸縮圖 + aspect-ratio 鎖 CLS；上線後以第三方 PSI 量測內頁 LCP/CLS，設內頁紅線。 |
| 段落生圖成本 | 每段必配 = 每段一次 OpenAI 生圖、按量計費（非 Claude 訂閱）；日更累積成本以較低生圖品質（`OPENAI_IMAGE_QUALITY=low`）+ webp 壓縮控管，張數不縮減。 |
| 段落圖機制尚未存在 | 需先擴 content schema + Markdown 渲染；列為實作前置相依（見 §6、§8）。 |

## 8. 交付項

1. 撰寫 `.claude/skills/newsroom/SKILL.md`（雷達複選 + 逐題問答 + 起草 + 超連結驗證迴路 + frontmatter 規則 + 繁中台灣用語）。
2. 段落配圖前置：擴 content schema / Markdown 渲染支援段落圖；段落圖效能處理（webp、縮圖、lazy-load、CLS 0）。
3. 串接生圖：封面圖 + 段落圖走 OpenAI gpt-image（沿用 `applyPeopleDirective` 台灣人物鐵律與既有金鑰）。
4. 本地預覽連結流程（`pnpm dev` 渲染 URL）。
5. 批次排程確認：依熱門度推薦 `publishDate`、列表確認、套用 + push。
6. 跑一次 end-to-end：雷達複選 → 逐題問答寫稿（含段落圖）→ 驗證 → 預覽修改 → 批次排程 → push → 線上確認；並以 PSI 複測內頁效能。
